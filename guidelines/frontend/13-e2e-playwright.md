# Frontend E2E with Playwright — Setup, Patterns, and Anti-Patterns

Use when adding Playwright to a Next.js project, writing the first E2E for a feature, or reviewing test code for the gate. Defines the project layout, the auth/storage-state pattern (log in once, reuse cookies), role-based selectors over CSS classes, API-driven seeding, and the **no `sleep`, no `waitForTimeout`** rule that keeps the suite deterministic.

Complementary to `qa/02-e2e-per-feature` — that guideline defines **what** must be tested (every new variant); this one defines **how** to write it cleanly in a Next.js + Playwright stack.

## Layout

```
frontend/
├── e2e/
│   ├── fixtures.ts            ← auth fixture, seed helpers, shared setup
│   ├── auth.setup.ts          ← runs once; saves storage state per role
│   ├── storage/
│   │   ├── admin.json         ← saved cookies for admin (git-ignored)
│   │   └── user.json
│   ├── datasets/
│   │   ├── archive.spec.ts
│   │   └── create.spec.ts
│   └── auth/
│       ├── login.spec.ts
│       └── password-reset.spec.ts
└── playwright.config.ts
```

- **One spec file per flow**, named after the user action.
- **`fixtures.ts`** centralizes the things every spec needs (custom `test` with a `login` fixture, factory helpers).
- **`storage/`** holds the cookie state per role — git-ignored, regenerated by `auth.setup.ts`.

## Playwright config — the essentials

```ts
// playwright.config.ts
import { defineConfig, devices } from "@playwright/test";

export default defineConfig({
  testDir: "./e2e",
  fullyParallel: true,                 // parallel workers
  forbidOnly: !!process.env.CI,        // .only ships only in dev
  retries: process.env.CI ? 2 : 0,     // CI retries for flake; dev does not
  workers: process.env.CI ? 4 : undefined,
  reporter: process.env.CI ? "github" : "list",

  use: {
    baseURL: process.env.E2E_BASE_URL ?? "http://localhost:3000",
    trace: "on-first-retry",           // trace artifact when a test retries
    screenshot: "only-on-failure",
    video: "retain-on-failure",
  },

  projects: [
    { name: "setup", testMatch: /.*\.setup\.ts/ },
    {
      name: "chromium",
      use: { ...devices["Desktop Chrome"] },
      dependencies: ["setup"],         // runs auth.setup before specs
    },
  ],
});
```

- **`fullyParallel: true`** is the default. A test that can't run in parallel with another is a sign of shared state — fix the state, don't serialize the suite.
- **`retries: 2` in CI only.** Don't retry locally; you'll mask flake in development.
- **`trace: "on-first-retry"`** keeps artifact volume down — only flaky tests generate traces.

## Log in once — reuse storage state

```ts
// e2e/auth.setup.ts
import { test as setup, expect } from "@playwright/test";
import path from "node:path";

const AUTH_FILE = (role: string) => path.join(__dirname, `storage/${role}.json`);

setup("authenticate admin", async ({ request }) => {
  const res = await request.post("/api/auth/login", {
    data: { email: "admin@test.local", password: process.env.E2E_ADMIN_PW! },
  });
  expect(res.ok()).toBeTruthy();

  // Save the storage state (cookies + localStorage)
  await request.storageState({ path: AUTH_FILE("admin") });
});

setup("authenticate user", async ({ request }) => {
  // ... same shape, different role
});
```

Then in a spec:

```ts
// e2e/datasets/archive.spec.ts
import { test, expect } from "../fixtures";

test.use({ storageState: "e2e/storage/admin.json" });

test("admin can archive a dataset", async ({ page }) => {
  await page.goto("/datasets/123");
  await page.getByRole("button", { name: "Archive" }).click();
  await page.getByRole("button", { name: "Confirm archive" }).click();
  await expect(page.getByText("Dataset archived")).toBeVisible();
});
```

Why this matters: **logging in via the UI in every test is the #1 source of E2E slowness**. A 50-test suite that logs in via the form 50 times spends most of its time on the login page. The API-driven setup runs once per role.

## Custom test fixture for ergonomics

```ts
// e2e/fixtures.ts
import { test as base } from "@playwright/test";

type Fixtures = {
  login: (role: "admin" | "user") => Promise<void>;
};

export const test = base.extend<Fixtures>({
  login: async ({ context }, use) => {
    use(async (role) => {
      await context.storageState({ path: `e2e/storage/${role}.json` });
    });
  },
});

export { expect } from "@playwright/test";
```

Then `await login("admin")` works in any test that needs a quick role switch mid-test.

## Selectors — role-based, not CSS

```ts
// good — role-based, resilient to CSS / className changes
await page.getByRole("button", { name: "Archive" }).click();
await page.getByLabel("Email").fill("user@test.local");
await page.getByText("Dataset archived").isVisible();

// bad — CSS class, breaks when a designer renames the class
await page.locator(".btn-archive").click();
await page.locator("#email-input").fill("...");
```

Priority order:

1. **`getByRole`** with name — closest to what an assistive technology sees; doubles as accessibility verification.
2. **`getByLabel`** for form fields.
3. **`getByText`** for assertions on visible content.
4. **`getByTestId`** as the **escape hatch** — when role/label/text aren't unique enough, add `data-testid` to the JSX and use it. Reserve for cases where the markup genuinely doesn't lend itself to a role query.

Avoid CSS selectors entirely in tests. They couple tests to implementation details that don't matter to the user.

## Seed via API, not UI

```ts
// good — seed via API, then drive UI
test("user sees their archived datasets in the archived tab", async ({ page, request }) => {
  const dataset = await request.post("/api/datasets", {
    data: { name: "test-archived", status: "archived" },
  }).then(r => r.json());

  await page.goto("/datasets?tab=archived");
  await expect(page.getByText("test-archived")).toBeVisible();

  // teardown
  await request.delete(`/api/datasets/${dataset.id}`);
});

// bad — set up state by walking through the UI
test("user sees their archived datasets", async ({ page }) => {
  await page.goto("/datasets/new");
  await page.getByLabel("Name").fill("test-archived");
  await page.getByRole("button", { name: "Create" }).click();
  await page.getByRole("button", { name: "Archive" }).click();
  // ... 10 more steps before the actual test
});
```

The UI is what's being tested; setting up state through the UI mixes setup with assertion and slows the suite ~10×.

## Hydration, navigation, and Next.js gotchas

- **`page.goto()` waits for `load` by default** — enough for most pages, but a Server Component page that does heavy initial fetching may need `await page.waitForLoadState("networkidle")` if assertions fail because the data hasn't rendered. Use sparingly — `networkidle` is more brittle than `load`.
- **App Router server actions** are POSTs; if you click a button that triggers one, wait for the resulting render: `await expect(page.getByText("Saved")).toBeVisible()`, not a manual sleep.
- **`prefetch={false}`** on test-irrelevant `<Link>`s can keep the network log clean (optional).
- **Hydration mismatches** show up as console errors. Add a Playwright listener and fail the test on console errors during the slice's first runs — they're real bugs.

## No `sleep`, no `waitForTimeout`

```ts
// bad — flake on slow CI
await page.click("button");
await page.waitForTimeout(2000);
await expect(page.locator(".result")).toBeVisible();

// good — Playwright auto-waits for the assertion
await page.getByRole("button", { name: "Search" }).click();
await expect(page.getByText("3 results")).toBeVisible();
```

Every `await expect(locator).toBeVisible()` / `.toHaveText()` polls until the condition is true or the timeout expires. Manual sleeps don't poll; they're either too long (slow suite) or too short (flake).

If you genuinely need to wait for something Playwright can't auto-wait on (e.g. a timer-driven UI update), prefer `await page.waitForFunction(() => ...)` over `waitForTimeout` — at least it's a real condition.

## Console-error guard

```ts
// e2e/fixtures.ts (extension)
test.beforeEach(async ({ page }) => {
  const errors: string[] = [];
  page.on("pageerror", (e) => errors.push(`pageerror: ${e.message}`));
  page.on("console", (msg) => {
    if (msg.type() === "error") errors.push(`console.error: ${msg.text()}`);
  });
  test.info().annotations.push({ type: "errors", description: () => errors.join("\n") });
});

test.afterEach(async () => {
  const errs = test.info().annotations.find(a => a.type === "errors")?.description;
  if (errs) throw new Error(`Page emitted errors:\n${errs}`);
});
```

A clean test run with five errors in the console is **not** a clean test run. Most "the new variant rendered but threw on hydration" bugs (see `qa/02-e2e-per-feature`) surface here. Project decides whether to fail on every error or only on hydration-class errors — but a console error budget of zero is the right default.

## Flake — treat as a bug

A flaky E2E:

1. **Quarantine** (`.skip` with a `// FLAKE: <ticket>` comment).
2. **File a ticket** to fix the underlying cause.
3. **Cap quarantine count** (e.g. ≤ 3 at a time) — beyond the cap, no new features ship until the suite is healthy again.

The opposite — "this test is flaky, just rerun until it passes" — undoes the entire purpose of the gate.

## Anti-patterns

- **`page.evaluate(() => window.__REACT_INTERNALS__)`** — tests the implementation, not the user. Rewrite to drive the UI.
- **Logging in via the form in every spec** — slow, brittle. Use storage state.
- **One mega-test that walks every flow** — when it fails, the failure isn't localized. Split.
- **Selectors via `nth-child`** — break on every layout change. Use role / text / label.
- **Asserting on internal IDs** — `expect(page).toHaveURL("/datasets/abc-123-uuid")` breaks every reseed. Match on what the user sees, not on the UUID.
